# ConoHa移行後 アーキテクチャ再設計レビュー

最終更新: 2026-03-16
対象: `blog_starter_20260315`（Next.js + Nginx + systemd, ConoHa VPS運用）

## 1. 目的
ConoHa VPS移行後の運用を前提に、以下を満たす設計へ段階的に再構成する。
- 可用性: 単一VPS構成でも復旧可能で、障害点が特定しやすい
- 保守性: 開発・運用フローが文書化され、属人性を下げる
- 安全性: セキュリティを「アプリ」「リバースプロキシ」「OS運用」の3層で担保する

## 2. 現状サマリ（観測ベース）
- 構成: `Nginx(80) -> Next.js app(3000)`, `systemd` 管理, `deploy.sh` で rsync 配備
- ガード: CSP等のHTTPヘッダは `next.config.mjs` で設定済み
- 運用: health endpoint と smoke-check があり、最小運用は成立
- 未完: 独自ドメイン + HTTPSが未完了（runbook明記）
- 課題: ハードニングと運用統制の水準が「最小動作」寄り

## 3. 設計原則（今回の再考で固定する方針）
1. 単一責務: アプリの安全、ネットワーク境界、OS権限管理を分離する
2. 変更容易性: 1回のデプロイで「設定・実装・運用手順」が矛盾しない
3. 再現性: 構築手順はコマンド化し、手作業を最小化する
4. 最小特権: 人・プロセス・ネットワークの権限を必要最小限にする
5. 段階導入: まずP0を固定し、P1/P2は監視値で拡張する

## 4. 脅威モデル（要点）
### 4.1 公開面
- HTTP運用中の盗聴・改ざん
- Host headerや不正リクエストによる想定外ルーティング
- 静的/メディア配信経由での不正ファイルアクセス

### 4.2 アプリ面
- Markdown由来のXSS混入リスク
- 意図しない拡張子・MIMEの配信
- 将来API追加時の認証欠落

### 4.3 運用面
- `deploy` ユーザー権限の過大化
- 障害時の手順不整合（runbookと実体が乖離）
- 秘密情報のローテーション不備

### 4.4 サプライチェーン
- Node/依存ライブラリの脆弱性追随遅れ
- Nginx/systemd設定変更のレビュー不足

## 5. ギャップ分析（ConoHa移行で見直すべき点）

## 5.1 基盤（VPS/OS/Nginx/systemd）
- G1: HTTPS前提設計が未完成
- G2: Nginxで `server_name _;` のまま（本番Host固定が未完）
- G3: セキュリティヘッダ責務がアプリ偏重（Nginx側の境界制御が薄い）
- G4: systemdハードニングパラメータ（NoNewPrivileges等）が未定義
- G5: fail2ban/ufwの運用手順はあるが、監査手順が未定義

## 5.2 設計（運用モデル/責務分離）
- D1: docs間の責務境界が弱く、変更時に追従漏れが起きやすい
- D2: 変更管理単位（アプリ変更と運用変更）のレビュー境界が曖昧
- D3: 障害対応の「判定基準（どこまでで復旧成功か）」が定量化不足

## 5.3 実装（コード/スクリプト）
- I1: `/media` の配信ポリシーが緩め（許可拡張子・ディレクトリ扱いなど）
- I2: デプロイスクリプトが単一経路で、フェイルセーフが不足
- I3: ハードコードIPを含む運用コマンドが残り、環境切替に弱い
- I4: セキュリティ回帰テスト（HTTPヘッダ、危険リンクなど）が不足

## 5.4 セキュリティ（統制/監視）
- S1: HTTPS強制（HSTS含む）の適用基準が未確定
- S2: 鍵・シークレットの棚卸し/ローテーション周期が未定義
- S3: ログ保全（保存期間・閲覧権限・インシデント追跡）が未定義
- S4: 脆弱性対応SLO（発見から修正まで日数）が未定義

## 6. 目標アーキテクチャ（単一VPSの安全最小構成）
### 6.1 論理構成
- Edge: `Nginx :443(TLS終端)` + `:80 -> :443` リダイレクト
- App: `Next.js :3000` をローカルループバック限定
- Supervisor: `systemd` で再起動・ログ・権限管理
- Host: `ufw` で `22/80/443` のみ公開, fail2ban 有効

### 6.2 責務分離
- Nginx: TLS, Host制御, request size/timeout, 入口レート制御
- Next.js: 画面/コンテンツ処理, アプリレベルセキュリティヘッダ
- systemd: プロセス権限最小化, 障害時再起動
- Scripts: デプロイ/診断/復旧の自動化

### 6.3 設計上の禁止
- 公開インターフェースを増やすときに認証/監査なしで出す
- 本番IPのハードコードを package scripts に残す
- 運用手順と実装差分をドキュメント未更新で放置する

## 7. 段階導入ロードマップ
### Phase 0（即時, 1週間）
- HTTPS完了、証明書自動更新確認
- 本番ドメイン固定（`server_name`）
- P0セキュリティテスト追加（health, headers, media policy）
- 運用コマンドのハードコードIP排除

### Phase 1（短期, 2-4週間）
- systemd/Nginxハードニング適用
- 監査ログ運用（保存期間、確認頻度、障害チケット連携）
- deploy手順に「失敗時ロールバック判定」を追加

### Phase 2（中期, 1-2か月）
- デプロイ方式の原子的更新（リリースディレクトリ方式）
- 監視拡張（CPU/MEM/Disk + エラーレート）
- 依存脆弱性対応SLO運用開始

## 8. 実装バックログ（優先度付き）

## 8.1 P0（必須）
1. Nginx HTTPS化完了 + HTTPから恒久リダイレクト
2. `server_name` を本番ドメインに固定
3. `/media` 配信ポリシー強化（許可拡張子、ディレクトリ拒否、`nosniff`）
4. `ops:*` スクリプトから固定IPを除去（`.env`/引数注入化）
5. セキュリティ回帰テスト追加（主要ヘッダ、危険URL、health）

## 8.2 P1（推奨）
1. systemd hardening の明文化と反映
2. Nginx側のレート制限・タイムアウト最適化
3. runbookに「復旧成功条件（SLO）」を明記
4. ログ保存ポリシー（期間・容量上限）導入

## 8.3 P2（拡張）
1. blue/greenに近いディレクトリ切替デプロイ
2. 監視通知（閾値超過時の通知先）
3. 監査証跡の定期レビュー運用

## 9. テスト戦略（設計）
- 単体: URL正規化/危険URL拒否/配信判定ロジック
- 結合: `/api/health`, `/media/*`, `/blog` のHTTP検証
- E2E: 本番相当URLでの smoke + headers
- 運用: `remote-status` を基準にした日次チェック

## 10. 受け入れ基準（Definition of Done）
1. HTTPSでの公開が完了し、HTTP直アクセスがリダイレクトされる
2. `publish:secure` + smoke-check が連続3回成功する
3. 主要セキュリティヘッダをE2Eで検証できる
4. 障害復旧手順がrunbook通りに再現できる
5. 変更内容が docs / ops / scripts / code で整合している

## 11. ヒアリング必須の保留タスク（今回は未着手）
- H1: 本番ドメイン最終決定（www有無、正規ホスト）
- H2: ログ保存期間と保全方針（個人運用か、監査要件ありか）
- H3: 侵入試行時の通知先（メール/Slackなど）
- H4: 障害時RTO/RPO目標（どこまでを許容するか）
- H5: 将来API公開範囲（認証方式の選定に直結）
