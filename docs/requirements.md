# ブログ要件定義（実装反映版）

最終更新: 2026-03-15
ステータス: MVP完成（運用開始可能）

## 1. サービスとして実現したいこと（確定）
- Markdown（`.md`）ファイルを追加すれば、ブログ記事ページとして公開できる
- 機械学習モデルのデモを記事内や専用ページとして掲載できる
- ホームページ上部にページ選択ナビゲーションを持つ
- 初回公開時のナビ項目は `home` / `blog` / `blog-tech`
- 将来拡張で `about` / `access` を追加できる構成にする
- Markdown 内の数式（`$$...$$`）を表示できる
- 画像を記事内で扱える

## 2. 目的
- 運用者（あなた）が将来的に機能を自由に拡張できるブログ基盤を作る
- 記事作成から公開までを短時間で回せるようにする
- デザイン/機能のカスタマイズを、コード編集で素直に反映できる構成にする
- コンテンツの判断（何を書くか、どう見せるか）は運用者が `.md` だけで決められるようにする

## 3. 非機能の優先事項（仮）
- 拡張性: 高
- 保守性: 高
- 表示速度: 高
- 初期開発速度: 中〜高
- 運用コスト: 中

## 4. 想定ユーザー（仮）
- 読者（一般ユーザー）
- 運用者（あなた）

## 5. 必須機能（MVP）
- `.md` ファイル投入で記事ページ生成（最優先）
- `home` コンテンツを `.md` で管理できる
- 記事一覧ページ
- 記事詳細ページ
- ホーム上部グローバルナビ（`home` / `blog` / `blog-tech`）
- 技術デモ掲載ページ（JSで動くデモ埋め込み）
- レスポンシブ対応
- 数式レンダリング（`$$...$$`）
- 画像表示対応（ローカル画像アセット）

## 6. 機械学習デモ要件（初期）
- デモは1ページに複数配置可能
- 初期MVPはブラウザで完結するデモのみ
- 将来拡張でAPI連携デモを許容する
- デモページはブログ記事とは別に管理できる
- 重い処理は遅延読み込みできる
- 将来、インタラクティブデモ（例: 手書き数字を推論するMNISTデモ）を追加できる

## 7. デモ拡張方針（確定）
- `.md` は文章コンテンツ管理に使う
- インタラクティブデモは `.md` とは別に「デモコンポーネント/デモページ」として実装する
- デモ一覧ページから各デモページへ遷移できる構成にする
- PC/スマホの入力（マウス/タッチ）に対応する
- モデル差し替え（ONNXやTensorFlow.js想定）がしやすい構造にする

## 8. プロダクト方針（確定）
- 多機能化は初期MVPで行わない
- まずは「少ない機能で綺麗に見える」ことを優先
- あなたは主に `.md` 作成だけで運用可能、コード変更は任意にする
- 将来の拡張（ページ追加、機能追加、デザイン差し替え）に備えた構造を採用する
- デザインは複数パターンを試せるようにする（初期は3案）

## 9. 拡張しやすさの設計方針（仮）
- フロントとデータ層を分離
- ドメインロジックを `features/` 単位で分割
- 依存の向きは `ui -> application -> domain -> infra` を基本にする
- テーマ/コンポーネントを差し替え可能にする
- テストを最低限（単体+E2Eの最小セット）整備する

## 10. 技術候補（保留）
### 案A（推奨）
- Next.js (App Router) + TypeScript
- 記事: Markdown（Git管理）
- 検索: ローカルインデックス（ビルド時生成）
- デプロイ: Vercel

メリット:
- 拡張しやすい
- 開発体験が良い
- ブログ機能とアプリ機能を同居しやすい

懸念:
- 将来の大規模CMS運用時は別途移行検討が必要

### 案B
- Astro + Content Collections

メリット:
- コンテンツサイトの表示速度が高い

懸念:
- 将来アプリ寄りの拡張を多くすると設計調整が増える

## 11. データモデル（最小）
- Post
  - `slug`（ファイル名から生成）
  - `title`（Markdown本文の先頭見出しから推定、未設定時はslug）
  - `content`
  - `publishedAt`（Git初回コミット日時から自動付与）
  - `updatedAt`（Git最終コミット日時から自動付与）
- Author（将来拡張用）

## 13. 決定事項と未確定事項
1. `.md` の配置ルール（確定）
   - `content/home.md`
   - `content/blog/*.md`
   - `content/blog-tech/*.md`
2. `blog` / `blog-tech` の振り分け方法（確定）
   - フォルダ分けで運用する
3. 公開フロー（確定）
   - 公開コマンド実行時点で、対象フォルダに存在する記事を公開対象にする
   - 作成途中は `content/wip/` フォルダで管理（非公開）
4. メタデータ運用（確定）
   - Frontmatter必須項目なし
   - 作成日/更新日はGit履歴から自動付与
5. 画像運用（確定）
   - `content/assets/` に配置し、Markdownから相対パスで参照
6. デザイン方針（確定）
   - 初期実装で3パターン作成し、比較して1つを採用
7. 公開目標日（未確定）
