# 掲示板機能 草案（管理人向け）

最終更新: 2026-03-20

## 1. 目的
- ブログ読者との継続的な対話導線を作る。
- 記事ごとの感想・質問を蓄積し、次の記事改善に活かす。
- 運用負荷と荒らしリスクを制御しつつ、段階的に機能拡張する。

## 2. 前提（現行構成）
- 現行は Next.js + Markdown中心の構成で、記事表示は静的寄り。
- 動的APIは `access-counter` が中心で、掲示板用の投稿CRUDは未整備。
- 本番は ConoHa VPS / `systemd` / `deploy.sh` 運用。

## 3. 実装方式の選択肢（比較）
### A. 外部埋め込み（Disqus/Giscus等）
- 概要: 外部サービス提供スクリプトを埋め込む。
- 長所: 最短導入、モデレーション/スパム対策が即利用可能。
- 短所: CSP調整が必要、外部依存とデータ主権の課題。
- 向くケース: 需要検証を最速で行いたいとき。

### B. BaaS（Supabase/Firebase）
- 概要: UIは自作し、DB/認証/ルールをBaaSに委譲。
- 長所: UI統一しやすい、RLSや認証の基盤が強い。
- 短所: ランニングコスト、ベンダーロックイン、移行コスト。
- 向くケース: 運用負荷を下げつつ中長期で拡張したいとき。

### C. 自前API + 自前DB（Next API + SQLite/Postgres）
- 概要: 既存Next APIルート上で掲示板APIを実装し、DBに保存。
- 長所: 制御性・独立性が高く、現行構成との整合が良い。
- 短所: モデレーション、バックアップ、監視責務を自前で持つ。
- 向くケース: サービス依存を減らし、運用方針を細かく制御したいとき。

## 4. 現時点の推奨方針
- 原則: 「小さく始めて安全に育てる」。
- UI形態: まずは独立掲示板より先に「記事ごとのコメント欄」から開始。
- 投稿形式: 初期はプレーンテキスト限定（Markdownは後段）。
- データ層: JSON直書きではなく SQLite を第一候補にする。

## 5. 段階導入計画（草案）
### Phase 0: 要件確定（1日）
- ゴール定義: 雑談/感想/質問のどれを重視するか。
- 運用ルール: 削除基準、通報対応、保存期間を決定。
- 公開方式: 即時公開か承認制かを決定。

### Phase 1: MVP（3-5日）
- 機能:
  - 投稿作成
  - 投稿一覧（新着順）
  - 管理者削除
  - 通報
- 制限:
  - 文字数上限
  - 連投制限（レート制限）
  - NGワード
- 対象:
  - 記事ページ下部のコメント欄

### Phase 2: 運用強化（1-2日）
- 管理機能:
  - 通報一覧
  - 投稿非表示/削除
  - 簡易BAN（IPハッシュ単位）
- 保守:
  - 日次バックアップ
  - 障害時復元手順の文書化

### Phase 3: 拡張（必要時）
- スレッド式掲示板化
- 投稿編集/リアクション
- ログイン導入（任意）

## 6. データモデル（最小案）
- `forum_posts`
  - `id`
  - `post_slug`（どの記事への投稿か）
  - `author_name`（任意）
  - `body`
  - `status`（visible/hidden/deleted）
  - `created_at`
  - `ip_hash`
- `forum_reports`
  - `id`
  - `forum_post_id`
  - `reason`
  - `created_at`
- `forum_bans`
  - `id`
  - `ip_hash`
  - `reason`
  - `expires_at`

## 7. 実装時の重要論点
- 保存方式:
  - 単一JSONの read-modify-write は掲示板用途で競合/性能面の限界が早い。
- セキュリティ:
  - レート制限、入力バリデーション、CSRF方針、通報導線が必要。
- XSS:
  - 初期はプレーンテキストのみ許可し、HTML/Markdown許可は後段評価。
- 運用:
  - バックアップと復元手順がない状態で公開しない。

## 8. 既存構成に対する変更方針
- 既存 `src/lib/content.ts` は記事責務のまま維持。
- 新規に `src/lib/forum/*` を追加し、動的機能を分離。
- APIを `src/app/api/forum/*` に分割し、責務別に拡張可能にする。
- UIは `src/components/forum/*` で分離し、既存Markdown処理に過剰な責務を載せない。

## 9. 未決事項（先に決めるべき3点）
1. 匿名投稿を許可するか。
2. 即時公開か承認制か。
3. 記事コメント型から始めるか、独立掲示板から始めるか。

## 10. リリース判断基準（MVP）
- 技術面:
  - `lint` / `test` / `build` が通る。
  - 主要導線（投稿・一覧・削除）が正常動作。
- 運用面:
  - 管理者が5分以内に不適切投稿を非表示化できる。
  - バックアップと復元手順を1回実地確認済み。
- 品質面:
  - 連投・NGワード・長文投稿の境界テストがある。

## 11. 結論
- 現行プロジェクトに最も整合的なのは「自前API + SQLite」の段階導入。
- ただし需要検証を急ぐなら、短期で外部埋め込みを試し、継続価値確認後に自前実装へ移行する二段構えも有効。
