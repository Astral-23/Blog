# 掲示板機能 草案（WordPress移行後）

最終更新: 2026-03-22

## 1. 目的
- 記事読者の反応を蓄積し、運用改善に使う。
- 荒らし対策と運用コストを制御しながら段階導入する。

## 2. 前提（現行）
- 配信は WordPress。
- 記事正本は Markdown (`content/*.md`)。
- 互換APIは `hutaro-bridge` が提供（health / access-counter）。
- 掲示板用の投稿CRUDは未実装。

## 3. 実装方式の選択肢
### A. 外部埋め込み（Disqus/Giscus等）
- 長所: 最短導入、モデレーション機能が即利用可
- 短所: 外部依存、データ主権の課題

### B. WordPressプラグイン方式
- 長所: WP管理画面と統合しやすい、運用導線が自然
- 短所: プラグイン保守責務、セキュリティ検証が必要

### C. 独立API + DB（WP外部サービス）
- 長所: 制御性・拡張性が高い
- 短所: 運用対象が増える

## 4. 推奨
- 先に A か B で小さく検証し、定着後にCを再評価。
- 初期は「記事コメント型」から開始し、独立掲示板は後段。

## 5. 段階導入
### Phase 0
- 投稿ポリシー（匿名可否、削除基準、通報運用）を確定

### Phase 1 (MVP)
- 投稿作成
- 投稿一覧
- 管理者非表示/削除
- レート制限

### Phase 2
- 通報一覧
- 簡易BAN
- バックアップ/復元手順

### Phase 3
- スレッド化
- リアクション
- 認証連携

## 6. データ最小案
- `forum_posts`: `id`, `post_slug`, `author_name`, `body`, `status`, `created_at`, `ip_hash`
- `forum_reports`: `id`, `forum_post_id`, `reason`, `created_at`
- `forum_bans`: `id`, `ip_hash`, `reason`, `expires_at`

## 7. セキュリティ必須要件
- 入力バリデーション
- XSS対策（初期はプレーンテキスト）
- レート制限
- 管理操作の監査ログ

## 8. リリース判断
- `lint/test/build` が通る
- 管理者が短時間で不適切投稿を非表示化できる
- 復元手順の実地確認が済んでいる
